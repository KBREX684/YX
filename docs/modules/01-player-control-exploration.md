# 玩家控制与探索模块

版本：v0.3.6
关联总设定版本：v0.8.6
状态：死亡复活工程口径接入
创建日期：2026-05-14
最后更新：2026-05-14

## 模块定位

玩家控制与探索模块负责副本内最基础的行动、视野、交互和生存状态。它是玩家接触所有恐怖反馈、线索、资源和目标路径的入口。

这个模块的核心不是让玩家强大，而是让玩家在有限能力下做出判断：什么时候走、什么时候躲、什么时候关灯、什么时候冒险互动。

## 设计目标

- 让玩家移动和交互足够清晰，避免操作本身成为主要挫败来源。
- 保持脆弱感，避免玩家通过高机动性或高战斗力压过恐怖体验。
- 支持 2.5D 场景下的遮挡、视线、光源和躲藏玩法。
- 为声音、心跳、手电异常等反馈预留表现空间。
- 让探索节奏在“慢速观察”和“短时逃命”之间切换。

## 核心功能

### 基础行动

- 行走：默认探索速度，噪音低，适合观察环境。
- 奔跑：快速转移，消耗体力，提高噪音和被注意风险。
- 蹲伏：降低噪音和可见性，移动速度降低。
- 停止呼吸或屏息：短时间降低被感知概率，但提高压力或消耗精神。
- 转向与视线控制：支持玩家检查角落、门缝、走廊尽头和关键物品。

### 躲藏交互

- 柜子、桌下、门后、床底、阴影区等躲藏点。
- 躲藏点需要有风险差异，例如柜子安全但可能被打开，阴影区快速但不稳定。
- 躲藏时玩家仍能接收声音、心跳、画面异常等反馈。
- 部分怪物可以学习玩家躲藏习惯或通过规则破坏躲藏点。

### 光源操作

- 手电开关。
- 手电照射方向和范围。
- 电量消耗与备用电池。
- 闪烁、变暗、短暂失效等异常反馈。
- 特殊情况下照出不可见痕迹、仪式符号或怪物残影。

### 交互对象

- 门、锁、机关、容器、文件、录音设备、仪式物。
- 交互需要有时间成本，重要操作可能暴露玩家。
- 高风险交互可被怪物靠近、光源异常或理智波动打断。

### 生存状态

- 体力：影响奔跑、屏息和挣脱。
- 理智：影响画面异常、误听、解谜信息可靠性和操作稳定性。
- 受伤状态：影响移动速度、噪音和互动效率。
- 污染值：与异常接触、原形携带和错误收容相关。

### 战斗、镜头与破坏边界

- 玩家没有直接攻击动作，不能像动作游戏一样主动普攻怪物。
- 对抗怪物主要依靠道具、环境、弱点条件、仪式、原形能力和副本规则。
- 第一阶段采用固定镜头，利用构图、遮挡、走廊纵深和房间切换制造压迫感。
- 躲藏点和关键交互物不会被永久破坏，避免单次错误让副本不可完成。
- 玩家死亡后从基地复活，并损失一定副本资源或带回收益。

## 输入与输出

输入：

- 玩家操作输入。
- 地图碰撞、躲藏点和交互对象信息。
- 怪物感知规则。
- 恐怖压力模块传来的异常状态。

### 第一阶段输入动作映射

P1-1 施工前必须在 `project.godot` Input Map 中定义以下 action，并且玩家脚本只能通过 action 读取输入：

| Action | 用途 |
|---|---|
| `move_left` / `move_right` / `move_up` / `move_down` | 2.5D 场景移动 |
| `run` | 奔跑 / 高噪声移动 |
| `crouch` | 蹲伏 / 低噪声移动 |
| `flashlight` | 手电开关 |
| `interact` | 开门、拾取、阅读、启动机关 |
| `hide` | 进入或退出躲藏点 |
| `pause` | 暂停 / 调试菜单 |

禁止在 `player.gd` 或状态脚本中硬编码物理按键；后续改键、手柄或键鼠布局都必须通过 Input Map 处理。

输出：

- 玩家位置、姿态、噪音等级和可见性。
- 当前交互状态。
- 体力、精神、伤势、污染等生存参数变化。
- 给怪物模块使用的玩家暴露信息。

## 与其他模块关系

- 向怪物与异常规则模块提供玩家噪音、光源、位置、姿态等判断条件。
- 从恐怖感知与压力模块接收心跳、画面异常、手电异常等表现。
- 与线索解谜模块共享交互系统。
- 与搜刮资源模块共享容器、拾取和背包规则。
- 与原形携带与助战模块共享随行、能力触发和保护逻辑。

## 第一阶段原型范围

第一阶段只需要完成最小可验证能力：

- 行走、奔跑、蹲伏。
- 一个手电系统，包含开关、电量和闪烁反馈。
- 两类躲藏点：固定柜子和临时阴影区。
- 基础交互：开门、拾取物品、阅读线索、启动机关。
- 三个生存参数：体力、理智、污染值。
- 微切片内先验证：走廊 + 2 个房间 + 1 个躲藏点中的移动、手电、互动、躲藏和噪声事件，再进入完整破败校园地图施工。

### P1-1 工程落地口径

- 玩家场景为 `scenes/player/player.tscn`，根节点 `CharacterBody2D`，挂载 `Visual`、`CollisionShape2D`、`StateMachine` 和 `Flashlight`。
- `StateMachine` 先使用 LimboAI 的 `LimboHSM` + `LimboState` 节点建立 `idle / walk / run / crouch / hide` 骨架；P1-1 只完成最小输入注入，后续可把复杂转移条件迁入可视化状态机。
- 手电参数由 `data/items/flashlight.tres` 驱动，字段包含电量上限、每秒消耗、低电量阈值、正常光强、低电量光强和关闭光强。
- 噪声事件统一走 `EventBus.noise_emitted(level, position, source_action_id)`，其中 `source_action_id` 必须是稳定 Input Map action 或后续资源 ID。
- 当前玩家美术为占位 `Polygon2D` + 场景内标签“占位: 玩家2.5D Live分层立绘”，对应后续需替换的正式玩家分层立绘资源。

### P1-3 交互物占位落地口径

- 第一阶段交互物统一为 `Area2D` 场景，最低接口为 `interact(player) -> Dictionary`，由 `scripts/objects/interactable.gd` 提供通用 payload 与 `interacted` 信号。
- `door` 负责开关和 `is_locked` 占位字段；`pickup` 返回 `item_id / amount`，由玩家控制器写入临时背包，P4 搜刮系统上线后替换；`note` 返回 `dialogic_timeline_id / note_text` 并发出 `EventBus.clue_collected`；`hiding_spot` 返回躲藏 payload，由玩家进入 `hide` 状态。
- Dialogic 2 在 P1-3 不启用运行时 Autoload，避免突破 5 个 Autoload 白名单；当前只保留 `dialogic_timeline_id` 作为 P3 线索/对话接入点。
- `abandoned_school.tscn` 已放入入口门、出口锁门、电池、规则纸条、躲藏柜 5 个实例，覆盖开门、拾取、阅读线索、进入躲藏点四类核心交互。
- 当前对象美术均为 `Polygon2D` + `PlaceholderAssetLabel`：破败校园木门/门锁分层、电池拾取物、纸条/线索文本、躲藏柜/阴影区。

## 第一阶段验收标准

- 玩家可以在一个完整副本中完成探索、躲藏、搜刮和目标交互。
- 奔跑、蹲伏、开灯、关灯都会影响怪物判断。
- 玩家能通过手电、声音和心跳变化判断危险接近。
- 操作失误会制造紧张，但不会让玩家觉得控制不可靠。

## 后续扩展方向

- 加入屏息、探头、门缝观察等高压动作。
- 加入受伤部位和行动负面状态。
- 加入不同角色或装备带来的行动差异。
- 加入原形改变玩家控制方式的特殊副作用。

## P3-4 死亡复活工程口径

P3-4 将"死亡后从基地复活"接入工程骨架：

- `EventBus.player_died` 由 `GameState` 订阅，死亡时调用 `respawn_at_base()`。
- `respawn_at_base()` 将高级场景状态切回 `SceneId.BASE`，清空 `current_dungeon_id` 与本次副本线索态，并发出 `EventBus.scene_changed("dungeon", "base")`。
- `base_placeholder.tscn` 是第一阶段基地占位场景，带 `placeholder_asset_note`；P5 替换为正式厚涂 2.5D Live 基地分层资产时保留复活入口契约。
- 死亡提示由 `DeathFeedbackResolver` 从关联 `RuleResource.learnable_hint` 读取；无法定位规则时显示 fallback，并把缺失 rule id 记录到 `GameState.missing_death_hint_rule_ids`。
- 当前 P3-4 只做占位资源损失计算：带入资源全损，副本内拾取返还率默认 0%；P4 搜刮系统会把拾取返还率接为 65%。

## 已确认设计决策

- 玩家无直接攻击动作。
- 第一阶段采用固定镜头。
- 躲藏点不会被永久破坏。
- 死亡后从基地复活，并损失资源。
